# 领域知识库 · 目标、上下文与方法

本文是本仓库 **团队领域知识库（Domain Knowledge）** 的人读标尺：说明「为何做、做到什么程度、如何切块」。

**开源默认**：第一节与通用方法已写好；**第二节业态必须由你们填写**（与 Cursor 对话，推荐 `@setup-domain-ops`）。业态切块 **不** 来自仓库预置模块表。

机器侧认域配置（`s2-domain-profiles.json`、`s2-propose-industry-seeds.json`）由 Cursor **根据已填实的第二节派生**，人确认后再跑 S1/S2。脚本 **不** 解析本文正文。

填写示例（虚构业态，勿当默认模块）：[`strategy.example.md`](strategy.example.md)。

---

## 一、目的

通过对 Confluence 各页面领域知识的梳理，形成 **可复用的团队领域知识库**，服务于敏捷交付与产品探索：

1. **新人上手**：缩短从入职到有效参与迭代的时间。
2. **需求与 User Story 的前置上下文**：识别歧义、边界与风险，**减少 sprint 内返工**。
3. **激发讨论**：连接「已有领域规则」与「潜在新价值点」。

### 总体取向：聚焦业务领域知识（规则清晰、提取全面）

- **聚焦业务领域**：篇幅优先给 **规则、流程、资格、约束与用户/业务对象可见后果**；协作排期、工单矩阵、纯运维链路等降级为溯源或弱化。
- **规则清晰**：条件、例外、分支、频率、时间窗用可扫读句式写清；术语与 [`language/glossary.md`](language/glossary.md) 对齐。
- **提取全面**：对 `materialized/` 中与业务相关的素材做穷尽式提取——凡影响「该不该、会不会产生可见后果」的条款须在 `curated/` 可核对；**不是**复制每句工程废话。

### 使用人群与可操作性（详略准则）

1. **业务/产品/运营（决策读者）**：是否可发布、是否合规、是否有冲突口径。
2. **需求/测试/交付（执行读者）**：落到 story、验收、缺口清单。
3. **开发/数据（实现读者）**：字段与接口如何映射业务语义，但不应反客为主。

三层信息密度：

- **L1（30 秒）**：范围、适用对象、关键频率/时间窗、主要后果、待定冲突。
- **L2（3–5 分钟）**：完整规则链（条件–分支–例外–后果），可转验收。
- **L3（按需）**：接口、字段、票据、历史讨论——仅溯源。

约束：`S5` 正文至少 L1+L2；每条规则应能回答「谁在什么条件下会看到什么结果」；缩写首现带业务含义。

### 知识重心

| 倾向 **强化** | 倾向 **弱化** |
|----------------|----------------|
| 统一语言、业务规则与例外 | 通用工程规范、无关工具链 |
| 流程与状态的 **业务含义** | 无业务语义的纯字段罗列 |
| 风险与歧义（业务视角） | 纯运维/部署细节 |
| 用户/业务对象的可见承诺 | sprint/retro、bug list、协作排期本身 |

### 定稿须包含「全面的业务规则」（非索引库）

**S2–S3** 只做认域与聚合；**S4** 工作稿与 **S5/S6** 定稿才承载规则全文。**不得**用「详见 `materialized/`」代替规则链。

### 成稿优先序（S4–S5）

与 [`distill-quality-bar.md`](distill-quality-bar.md) 一致：

1. **语义重挂载** — 按领域命题组织正文。
2. **去伪存真** — 弱化协作/工程噪声，保留 Confluence 溯源。
3. **全面（业务侧）** — 影响可见后果的条文写全、可核对。
4. **语言** — **S1–S6 禁止翻译**；目标读者语言仅在 **S7**（见下节）。

### 合并准则

- 一条规则只讲一个判定问题。
- 按业务触发合并，不按页面标题合并。
- 同词同义、异词异义；冲突不静默覆盖。
- 先主线后支撑（实现口径进溯源）。

### S5 强制字段模板（机器门禁）

每条 `### 规则簇` 须含：`规则要点`、`适用对象`、`条件`、`处理分支`、`时间窗`、`例外`、`用户可见后果`、`参考来源`。

### 版式：尽量少用 Markdown 表格

默认用小标题 + 列表 + 短段落；仅在强对齐对照时用简表。版式服务内容，勿削足适履。

### 核心能力：Cursor 侧「提取、整理、去伪存真」

流水线同步草稿；**定稿判断**在 Cursor Agent。脚本 **不** 调用云端 LLM 写 `curated/`。工序见 [`generate-knowledge-from-wiki/RUNBOOK.md`](../.cursor/skills/generate-knowledge-from-wiki/RUNBOOK.md)。

### 语言：S6 原语言定稿 · S7 目标语言

| 阶段 | 路径 | 语言 |
|------|------|------|
| 同步 | `extracted/`、`materialized/` | Confluence 原文 |
| 认域+聚合 | `_aggregate/`、closure、清单 | 不翻译 |
| 蒸馏工作稿 | `_deliver/…-工作稿.md` | 与源一致 |
| 原语言定稿 | `_deliver/…-source-brief.md`（**S6**） | 与源一致（裁决完成，不翻译） |
| 目标语言定稿 | `_deliver/…-领域知识定稿.md` / `…-domain-brief.md`（**S7**） | `deliverable_locale`（常见 zh-CN / en） |

## 二-附、从 strategy 到 profiles（工序）

1. **填实第二节**（人 + Cursor）。
2. Cursor 根据第二节起草：
   - [`s2-domain-profiles.json`](s2-domain-profiles.json) → `checklist_themes`、`s1_facets`、`s2.primary_facet_to_slug`、`domain_cues` 等
   - [`s2-propose-industry-seeds.json`](s2-propose-industry-seeds.json) → `module_seeds`（可按团队 `teams: ["<key>"]`）
   - 可选：[`jira/teams/<key>.json`](jira/teams/demo.json) 的 classify facets
3. **人确认** slug / 轴心 / 关键词后落盘。
4. 再跑 `@generate-knowledge-from-wiki`（S1 → S2…）。

若 `checklist_themes` 仍为空，认域脚本与 wiki skill **硬停**，并提示回到本工序。

## 四、多团队 / 多产品线

- 每个权威根 → `extracted|materialized|curated/by-root/<root_id>/`
- 团队元数据：[`jira/team-roots.json`](jira/team-roots.json)（任意 N 个 key）
- 入口：`@generate-knowledge-from-wiki` + Confluence 链接 → 备料 → 人标 `确认` → 成稿

**S1 facet 归类**只读 `s2-domain-profiles.json`（由本节派生），**不是**仓库内置业态。未命中 → `facet-unmatched/`。

## 五、子树根与父级根

| 做法 | 范围 | 目录 |
|------|------|------|
| 传子页面为根 | 该页及后代 | `by-root/<子页ID>/` |
| 传父页面为根 | 父及全部后代 | `by-root/<父页ID>/` |

实践：每个产品线选定 **一个权威父根** 作为完整领域库入口；子根适合增量，不替代权威根。

## 七、整理稿偏「乱」时（通用）

常见成因：源页本身是实现说明；抽取保留原文与破碎表；同树下材料质量不均。

减负：Cursor 整理时 **业务规则写全、技术噪声降级**；Confluence 侧尽量区分规则页与对接页；对已生成稿可定点「按 strategy 重写」。

---

## 八、第一性原理：蒸馏质量

**业务规则的高质量 = 可争论命题的完整、无歧义陈述**——依赖语义判断，脚本只能改善输入形态与做确定性校验，不能替代 Cursor + 人。

脚本可做：伴生清单、closure 覆盖检查、启发式质量报告。脚本不应假装「更强正则 = 领域真理」。

## 九、做完的定义（三档）

| 读者友好名称 | 含义 |
|--------------|------|
| **闭环** | `materialized/` 参与验收的页在 `_materialization_closure.json`（或同名路径）有落点 |
| **定稿（路径级）** | 单文件 S7 目标语言定稿可溯源（且有对应 S6 原语言稿） |
| **核心主题已定稿** | 确认页 `确认` 行均完成 S6 |

确认页状态 `确认` = 认可领域块划分并授权成稿。对外只认 **S7** 目标语言定稿（`*-领域知识定稿.md` / `*-domain-brief.md`）；其上游须有 **S6** `*-source-brief.md`。

展示层：[`TEAM_KNOWLEDGE_BASE.md`](../TEAM_KNOWLEDGE_BASE.md)。覆盖：`scripts/distill/coverage.py`。
