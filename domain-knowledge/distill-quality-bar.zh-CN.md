## 三层底线（产物 · 完成 · 节奏）

3. **节奏**：默认 `@generate-knowledge-from-wiki` 完成 **备料（S2）** 并 **暂停**；人标 **`确认`** 后 **`继续`** 跑 **成稿**（S3→S7，可按 slug 分批）。**禁止**仅有 `_aggregate`、领域模型、工作稿或 S6 却汇报「目标语言定稿完成」。

## 目标函数（优先序）

与 [`strategy.md`](strategy.md) §一「成稿优先序」一致；**本节为 S4–S6 执行裁判**（优先序、反目标、细则下文展开）。

1. **领域模型化**：先分层建立一等业务对象、指标/字段、展示容器、对象关系、状态机/状态转换、边界候选，再挂载规则链；**错误**目标是直接堆规则清单、把指标/字段/页面当对象，或保留 Confluence 小节外貌。
2. **语义重挂载**：按领域命题（资格—分支—后果、可见口径、时间窗）组织；**错误**目标是「少删字、保留 Confluence 小节顺序与标题外貌」。
3. **业务噪声隔离**：协作排期、纯工单矩阵、邮箱墙、DDL/SQL/Git、无业务解释的机械字段海 → **从主决策链分流为支撑材料/噪声，不让它们冒充业务规则**。
4. **全面（业务侧）**：凡影响「该不该、会不会对用户/顾问产生可见后果」的条文 **写全、可核对**；合并重复，**不**用「详见 `materialized/`」代替整条规则链。
5. **语言**：**S1–S5 禁止翻译**；**仅 S6** 产出读者向简体中文（专有名词、字段名可保留英文）。

### 可读性优先（最终生成物）

- S6 主文必须“业务叙事优先”：先给业务读者范围、模型摘要和可读规则，再给待定分类与溯源。
- Decision Atom / Conflict Ledger 是机器与审阅层，不替代主文阅读体验。
- 禁止把原子字段直接平铺到正文，导致“可校验但不可读”。
- S6 的领域模型摘要必须先分层：一等业务对象、对象关系、状态机/状态转换用粗体标签和缩进子项表达，避免多个对象或状态挤在同一行。
- S6 的每个核心规则必须是读者决策卡片，稳定包含 `已确认规则 / 待确认边界 / 用户可见影响 / 关联待确认事项`；每个标签必须作为独立粗体列表项，具体内容放入缩进子项，不得让读者从散文或长句里自行拼出规则边界。
- S6 的 `已确认规则` 不得用单个一级 bullet 加分号压缩多个分支、阶段、目标、奖励或展示后果；凡读者需要按分支判断的内容，必须拆成分层子项或放入 `关键明细` 的可查结构。
- S6 的 `待确认事项` 必须是行动索引，不是备注列表；每条待确认事项必须以 `影响规则` 作为主列表项，并在缩进子项中包含 `待确认/待补充 / 建议确认人 / 确认后影响`。
- S6 必须包含 `概述与范围 / 不在本文展开 / 领域模型摘要 / 核心业务规则 / 术语说明 / 待确认事项 / 溯源`。
- S6 不得把 S5 边界候选、噪声、工程协作、发布/合规支撑升格为核心业务规则。
- S6 不得把体验版、bug list、sprint/retro、代码提交频率、三方协作、项目启动/完成日期或阶段验收节奏升格为核心业务规则；这类材料只能作为交付边界、待确认事项或溯源证据。若它们确实影响用户可见质量，必须改写为“影响哪个业务对象的可见结果”，而不是保留交付流程本身。
- S6 不得出现 `$1`、`TODO`、明显截断验收句、重复“用户可见用户可见”等模板残留。
- S6 主文必须中文为主；凡在 `术语说明` 声明为 `中文术语（English Term）` 的关键业务术语，如果正文使用该中文术语，正文首次出现必须采用同样的 `中文术语（English Term）` 锚点，后续可只用中文。必要英文只允许作为括号内来源原名、字段/API/系统名、业务专有缩写或来源链接/路径出现；字段/API/系统名必须用反引号作为锚点，业务专有缩写必须在 `术语说明` 解释。
- S6 读者语言精修由 `domain-knowledge/language/s6-reader-language-policy.json` 定义通用表达类别；脚本只做确定性检查，禁止把具体业务域知识写进语言门禁。
- S6 必须保留来源中的结构化业务明细：映射表、状态枚举、字段清单、公式/阈值表、时间窗表、可见文案清单不得被摘要吞掉。若这些明细影响规则路径或用户可见判断，S6 应增加 `## 关键明细`，按业务规则分 `###` 小节，用表格或缩进列表保留可查询内容；若来源不足以形成稳定明细，则必须在 `待确认事项` 中写清待补内容与补齐后影响。

### 合并质量底线（清楚、合理、不混淆）

- 同一规则簇仅承载一个业务判定问题；不同判定问题必须拆分。
- 只有在“主体 + 触发条件 + 时间窗 + 可见后果”同源时才允许合并；任一关键轴不同则拆分。
- 冲突口径不得隐式吞并：必须显式写“采用规则”或“待确认事项”。
- 合并后正文应保持“先规则主线、后实现支撑”，避免接口字段压过业务语义。
- 命名业务结构是一等语义对象：源材料中的命名分支、阶段、路径、档位、等级、状态、周期、市场、卡片等，若承担业务判定或用户可见影响含义，S4/S5 必须保留其业务含义并写入领域模型、主链或待裁决区。
- 禁止把命名业务结构抽象成无名条件后消失；可以归一命名，但不得丢失读者用于判断规则路径的名称。
- 禁止把可查询的结构化明细改写成“若干字段”“多种状态”“相关映射”等概括性表达；S6 可以产品化语言，但不能牺牲业务读者按编号、状态、字段、时间窗查询答案的能力。
- S3 必须标记表格型来源信号：当 S1 有长映射表、字段表、状态枚举表、资格/等级/奖励档位表、时间窗表或可见文案表时，S3 页面级 `structured_source` 必须记录结构信号、表头/字段、密集行数量和样例行；这是脚本可做的结构保真，不是业务裁决。
- S5 必须承接结构化明细转交：当 S3 proposition、S3 页面级 `structured_source` 或 S5 语义重挂载发现映射表、状态枚举、字段清单、公式/阈值表、时间窗表、可见文案清单或编号/档位/等级映射时，工作稿必须写 `## 结构化明细转交`。S5 必须先按“完整展开 / 规则化压缩 / 待确认”裁定粒度，再按业务规则分小节保留可查询表格或分层列表。S6 的 `## 关键明细` 只能来自此区，不得越级回看 S1 补明细。
- S5 明细粒度三分法是验收标准：读者需要逐项查询或逐项判断、且每行可能改变身份/资格/奖励/状态/字段/时间窗/文案/例外时，必须完整展开；来源可由稳定公式、连续编号、等差阈值或同构命名无损生成，且单行没有独立例外含义时，才可规则化压缩，并必须写范围、公式/步长、端点、例外和示例；来源冲突、不完整、字段含义未闭环、是否用户可见未确认或展示位置不稳定时，必须列为待确认，不得用完整表格制造伪确定性。
- 排除材料、工程协作、发布/合规支撑、待迁移主题、待归属材料不是领域对象；不得放入 `## 领域模型` 的 `领域对象`，只能作为输入处置、边界待裁决或排除说明。
- 交付协作材料不是领域对象：体验版、bug list、sprint/retro、代码提交频率、三方协作、项目日期、阶段验收节点不得作为一等业务对象或核心规则；只允许降层为展示质量支撑、交付边界或待确认事项。
- 半闭环链与待裁决问题必须互链一致：链引用 `问题 N` 时，`问题 N` 必须存在并反向绑定该链；待裁决问题还必须写清 `决策影响`。
- `## 组织顺序说明` 是链序唯一真相；其它小节只能摘要说明顺序归一原则，不另写一套详细链序。
- `## 领域模型` 必须分层：一等业务对象、指标/字段、展示容器、对象关系、状态机/状态转换、边界候选分别写；指标、进度、公式、金额字段、API token、页面卡片不得混入一等业务对象。
- 闭环链必须挂到分层模型：链内 `领域对象` 必须来自 `一等业务对象`；指标、字段、公式、金额、进度、目标、页面、卡片、列表、API 只能进入 `展示容器/字段锚点`，不得作为链领域对象。
- S4 模型门禁由 `scripts/distill/domain_model_quality.py` 执行：只验证分层模型、链对象继承和字段/API/页面降层，不生成 S4/S5 正文。
- S5 工作稿门禁由 `scripts/distill/s5_work_draft_quality.py` 执行：只验证闭环链字段、组织顺序、待裁决互链、未决词位置和结构化明细形态，不生成 S5 正文。

### 默认：能整理成业务规则链条就写全

- **只要**素材能整理成 **`strategy.md` 行业模板**下的 **业务规则链条**（主体 / 条件 / 分支 / 时间窗 / 可见后果，且可溯源），**本路径的整理稿应直接写全该链条**，**不**为省事改用 Pass 占位。  
- **`facet-unmatched/`** 只是脚本侧的物化文件夹名，**不**等于「允许一律 Pass」；仍须 **逐页** 判断是否存在业务链条。  
- **S2** 认域打标；**S3** 仅对已确认块归集；**S4–S6** 按主题分批。**仅当**整页明显非业务时用 **Pass**（见下文），**不**用 Pass 回避可在 **S4/S5** 写出的模型与规则链。

## 落稿前分流（必备）

将素材划为三类再写字：

| 类 | 处理方式 |
|----|----------|
| **规范层** | 会员/顾问可见后果、阈值、分支、文案口径 → **业务规则**等语义章，分点写清。 |
| **映射与呈现** | Gateway/Path/字段、界面数据规则 → **数据与接口 / 呈现要点**，结构化列表为主；少用大表。 |
| **协作与运维噪声** | 不占正文主篇幅。 |

S3 保真分轨契约（脚本强约束）：

- 页面级必须标注 `doc_intent`：`rule_spec` / `api_contract` / `release_change` / `test_ops` / `discussion_decision`。
- 页面级必须标注 `structured_source`：长表/映射/枚举/字段清单即使单行命题弱，也必须留下结构信号与样例，供 S5 判断是否需要 `结构化明细转交`。
- 命题级必须标注 `candidate_type`：
  - `contract_candidate`：进入 `decision-atoms` 最小审计视图；S4/S5 Agent 仍负责领域模型与业务语义重挂载
  - `evidence_note`：保留证据与业务语义锚点，供 S4/S5 延迟裁决，不直接进入决策主链
  - `noise_context`：隔离无业务决策价值噪声，默认不进入 S4/S5 主链
- 每条命题必须包含 `eligibility_reason`，且保留 `scope/evidence_span` 溯源锚点。
- 每条命题必须包含 `semantic_roles` / `semantic_preservation_reason` / `business_scope_label`，用于证明 S3 没有静默丢语义。
- 每条命题必须包含 `decision_track`（`decision_core` / `presentation_context` / `unresolved_critical` / `noise_context`）与 `causality_score`。
- `decision_core` 才能作为 S5 主决策链输入；`unresolved_critical` 必须显式进入“待裁决关键问题”；`presentation_context` 只提供呈现/背景证据。
- S3 禁止根据单个样例、阈值大小或局部关键词推断业务分支；只保留源材料显式结构，领域模型与分支含义由 S4/S5 Agent 重挂载。

高置信触发原则（门禁策略）：

- 对 `doc_intent` 相关“比例类”门禁，采用高置信触发（标题强匹配 + 样本量阈值）后才硬失败。
- 禁止把启发式意图识别误差直接放大为全链路失败。
- 禁止通过下调总门槛掩盖分流问题，优先修复分类触发条件与候选资格规则。

因果闭环优先原则（新增）：

- 质量评估优先看“条件→分支→后果”是否闭环，而非仅看字段填充率。
- 对出现 `未确定/待确认/TBD` 的高信号命题，必须进入 `unresolved_critical`，禁止静默并入展示类上下文。
- `presentation_context` 可保留但不得挤占决策主链比例。
- 单阶段准入必须可审计：每条命题需携带 `admission_stage1_result`；`noise_context` 必须有 `admission_drop_reason`。
- 反误杀门禁：高因果命题（`decision_core/unresolved_critical` 且 `causality_score` 达阈值）若大量未进入 contract，S3.5 应硬失败。

## S2–S6 与「确认」（与 RUNBOOK 一致）

- **S2**：领域模块确认页 + **全文件打标** + `_materialization_closure.json`；多主题引用同一源 → **只索引，禁止多目录贴原文**。
- **S3**（**仅 `确认` 行**）：`_aggregate/<slug>/` 聚合；**不**要求中文、**不**要求叙事定稿。
- **S4**：`_deliver/<slug>/<slug>-领域知识-工作稿.md` 的 `## 领域模型` — 领域对象、状态/阶段、业务动作、状态变化、展示容器/字段锚点，**语言不变**。
- **S5**：`_deliver/<slug>/<slug>-领域知识-工作稿.md` — 基于 S4 模型挂载规则链、语义重挂载、业务写全、**语言不变**。
- **S6**（同上主题）：`_deliver/<slug>/<slug>-领域知识定稿.md` — **简体中文**读者向；**对外默认只读本文件**。
- **初稿 / 同步稿的引用方式**：**仅**在 **`## 溯源`**（或同等小结节）中保留 **最简形式** 之一：  
  - 权威页：**Confluence 链接**（主溯源）；  
  - 机器物化层（可选 **一行**）：`domain-knowledge/materialized/by-root/<根>/<相对路径>.md` 或仓库内 **单一路径** 文字，**不**在正文展开 `materialized/` 内容。  
- **同层互见**：`curated` 篇与篇之间用 **短横线或「见同目录某篇」** 指代 **其它定稿** 可接受；**不要**与「引用初稿」混为一谈。

**门禁脚本**：`scripts/distill/quality.py` 对 ``domain-knowledge/materialized/`` 的检测 **仅针对首个 `## 溯源` 标题之前的正文**；溯源区内允许 **一行** 初稿路径（与「禁止在正文链回 `materialized/`」一致）。

## 禁止项（反模式）

- **禁止**用一连串 **`### [Confluence 章节标题](url)`** 当作正文主目录（把来源结构误当领域结构）。
- **禁止在正文主文**（`## 概述`…`## 待定` 等主体）用 Markdown **链回** **`domain-knowledge/materialized/`** 或贴 **同步稿原文** 代替定稿；**`materialized/` 仅许出现在 `## 溯源` 下的一行路径**（见上节）。**S6 定稿**另须去掉「骨架/待补」类元语言。
- **禁止**对应有 `materialized/` 物化路径却 **不写** `curated` 同路径文件（Pass 也必须有着落）。
- **禁止**仅凭 **文件名模式** 或 **`facet-*` 目录名** 认定「模块已覆盖」——**S2** 须结合 **`strategy.md` 命题标尺** 与 **`materialized/` 全文检索** 补全 closure/清单路径；上一轮稳定 `glossary.md` 只能作为同义词/缩写提示，不得作为领域边界依据；**S4–S5** 再建立模型并合并正文。同一业务概念分散在多页时尤须如此。
- **禁止**用模板句冒充规则链（例如「进入业务场景即生效」「以后续实现为准」）；这类句子不提供可执行决策信息。
- **禁止**仅出现“条件/后果”标签却无具体锚点（阈值、状态迁移、时间窗、API/字段 token、枚举值）。
- **禁止**缺失 S6 强制字段：`适用对象/处理分支/时间窗/例外/参考来源`（缺失即视为未定稿）。
- **禁止**把“不同时间窗/不同资格口径”的条文合并成同一规则簇，造成读者误判。
- **禁止**在同一篇内对同一术语使用多个未声明含义（如“资格”“达标”“审核期”）。

## Pass（明显非业务）— 窄口径

仅当 **整页** 属于纯协作/运维/工程过程、**且** 不包含可整理为业务规则链条的素材时使用 Pass。  
仍须 **同路径** 写入极简稿：**`## 非业务判定（Cursor）`** + **`## 溯源`**；**禁止**空白文件；**禁止**用 Pass **回避**本可写出的规则链条。

## 脚本可做的自检（非 LLM）

同步或蒸馏迭代后可运行：

```bash
# 溯源闭环：每个 materialized/*.md 在 curated 有落点——优先 _materialization_closure.json（领域重组），
# 否则默认同名相对路径（兼容旧树）；可用 --exclude-prefix 暂抑部分前缀
python3 scripts/distill/coverage.py --root-id <根页ID>

# 命题中间层：确认主题必须有逐页命题清单（S3.5）
python3 scripts/distill/proposition_extract.py --root-id <根页ID>
python3 scripts/distill/proposition_quality.py --root-id <根页ID>

# 通用语义层：Decision Atom + Conflict Ledger（S3.6）
python3 scripts/distill/decision_atom_sync.py --root-id <根页ID>
python3 scripts/distill/conflict_ledger_sync.py --root-id <根页ID>
python3 scripts/distill/decision_atom_quality.py --root-id <根页ID>
python3 scripts/distill/conflict_ledger_quality.py --root-id <根页ID>

# 启发式：镜像骨架 / 正文链回 materialized 等（Pass 稿自动跳过深度规则）
python3 scripts/distill/domain_model_quality.py --root-id <根页ID>
python3 scripts/distill/s5_work_draft_quality.py --root-id <根页ID>
python3 scripts/distill/quality.py --root-id <根页ID>

# 领域优先排版：业务小节须在「实现/接口」重章之前（Pass 与小文件可跳过）
python3 scripts/distill/domain_layout.py --root-id <根页ID>
```
