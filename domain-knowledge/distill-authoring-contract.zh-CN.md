# S5/S6 作者契约

本契约定义 Compose 阶段 S5/S6 的最小落稿形态。它是跨领域约束：Agent 负责业务语义判断，脚本只验证这些判断是否以稳定、可审阅的 Markdown 结构显式表达。

## 为什么需要这份契约

S4/S5/S6 的质量失败通常不是 S1/S3 证据缺失，而是生成约束分散，导致 Agent 明白原则却没有按门禁需要的固定表达落稿。根因修复是把必需的作者界面集中说明：

- S5 必须显式表达模型挂载、业务顺序、待裁决互链和结构化明细。
- S6 必须把 S5 转成读者向中文知识产品，不新增也不丢失业务语义。
- 领域专属内容只进入生成正文，不进入本契约。

## S5 工作稿：章节顺序

每篇 `*-领域知识-工作稿.md` 必须使用以下顶层顺序：

1. `## 概述与范围`
2. `## 输入处置摘要`
3. `## 领域模型`
4. `## 组织顺序说明`
5. `## 已闭环决策链`
6. `## 待裁决关键问题`
7. `## 结构化明细转交`（当 S3/S5 存在可查询结构化明细时）
8. `## 溯源`

`## 概述与范围` 对工作稿也强制要求，因为 S5 已经是领域知识工作稿，不是原始实现台账。

`## 溯源` 必须列出 S3 高贡献、高价值来源的 Confluence URL 或等价来源链接；正文只保留来源标题或语义证据，来源链接集中放在溯源区。

## S5 输入处置摘要

`## 输入处置摘要` 必须显式提到并处置三类 S3 输入：

- `contract_candidate`：
  - 说明进入闭环链、拆成半闭环、进入待裁决，或被排除的原因。
- `evidence_note`：
  - 说明高价值证据是否被提升为闭环链、拆成待裁决、保留为结构化明细，或因明确原因被排除。
- `noise_context`：
  - 说明被排除、降层为边界/支撑证据，或作为待确认边界记录。

同时必须说明：

- 语义归一：重复、跨页面、跨 UI、跨 API 信号如何合并、降为字段锚点或拆开。
- 来源/顺序归一：来源顺序是否代表业务顺序；若不代表，如何转为业务判定顺序。
- 跨主题转交：若 `_aggregate/CROSS_SLUG_HANDOFF.md` 中存在指向当前 slug 的页面，必须说明该页面已被吸收、降为字段/展示/边界证据，或因明确原因排除；不得因来源目录属于其它 `facet-*` 而静默忽略。

不要在这里写详细链序。详细链序唯一来源是 `## 组织顺序说明`。

## S5 领域模型

`## 领域模型` 必须分层：

- 一等业务对象：能承载规则的业务对象。
- 指标/字段：公式、进度、金额、目标、API 字段、枚举值、阈值等。
- 展示容器：页面、卡片、列表、弹窗、API 响应、通知载体等。
- 对象关系：归属、依赖、承载、计算、展示、生命周期或前置关系。
- 状态机/状态转换：状态、资格、阶段、等级、可见性或金额变化。
- 边界候选：排除、待迁移、交付协作、实现支撑或不确定材料。

闭环链的 `领域对象` 只能来自 `一等业务对象`。字段、公式、页面、卡片、API、进度指标必须写入 `展示容器/字段锚点`。

## S5 组织顺序说明

链标题使用纯文本，不加粗，不使用代码样式。

```markdown
## 组织顺序说明

- 链 1：<必须与闭环链 1 标题完全一致> — <为什么排在第一>
- 链 2：<必须与闭环链 2 标题完全一致> — <为什么接在后面>
```

链号和标题必须与 `## 已闭环决策链` 完全一致。

## S5 闭环链块

每条闭环链必须使用以下形态：

```markdown
### 链 N：<标题>

- 领域对象：<必须列在一等业务对象中>
- 状态变化：<状态/资格/金额/可见性/阶段变化；确实没有时写“无状态变化，仅展示/校验/记录变化”>
- 业务动作：<业务动作或业务判定>
- 展示容器/字段锚点：<模型中的展示容器或字段；确实没有时写“无直接展示容器，来源仅确认后台判定”>
- 适用对象：<谁>
- 触发条件：<何时/阈值/状态/时间窗>
- 分支或动作：
  - <一个分支或动作>
  - <另一个分支或动作>
- 用户可见影响：
  - <一个可见后果>
- 关联待裁决：问题 N
- 证据来源：
  - <来源标题或 URL>
```

规则：

- `未确定`、`待确认`、`待定`、`unknown`、`TBD` 等未决词不得留在闭环承诺中，必须拆到 `## 待裁决关键问题`。
- 如果闭环链提到实现、API、字段、证据或检测方式缺失，必须写 `关联待裁决：问题 N`。
- 交付/协作语境不得成为闭环业务规则；除非直接改变用户可见业务承诺，否则只能降层为边界、支撑证据或待裁决问题。

没有闭环链时使用：

```markdown
## 已闭环决策链

当前未形成已闭环决策链。原因：<当前来源为什么不能支持业务承诺>。
```

## S5 待裁决问题块

每个待裁决问题都必须可双向绑定：

```markdown
### 问题 N：<问题标题>

- 关联链：链 N
- 关联前置域：<没有闭环链或问题发生在链之前时使用>
- 待裁决点：<需要裁决什么>
- 当前证据：<来源已经证明什么、没有证明什么>
- 待确认事项：<需要补什么或问什么>
- 建议确认人：<业务/产品/数据/工程等角色>
- 决策影响：<确认后会改变哪条规则、范围、字段、状态、展示或用户承诺>
```

闭环链引用了某问题时，问题块必须写 `关联链`。只有还不能绑定到具体闭环链时，才使用 `关联前置域`。

## S5 结构化明细转交

当 S3 proposition、页面级结构化来源信号，或 S5 语义重挂载发现表格、枚举、字段列表、映射、公式、阈值、时间窗、可见文案、编号、档位、等级、职称或奖励档位，并且读者可能查询时，必须出现 `## 结构化明细转交`。

每个 `###` 小节必须使用表格或分层列表。只有段落说明的小节不合格。

表格形态：

```markdown
### <明细标题>（完整展开）

| 项 | 条件/字段/状态 | 业务含义 | 例外/待确认 | 来源 |
|---|---|---|---|---|
| <行标识> | <来源明细> | <判定含义> | <例外或无> | <来源> |
```

分层列表形态：

```markdown
### <明细标题>（规则化压缩）

- 范围：<起止或包含集合>
  - 生成规则：<公式、命名模式、步长或分组规则>
  - 端点与例外：<端点和例外>
  - 示例：<一到两个示例>
  - 来源：<来源>
```

`待确认` 也必须用表格或分层列表说明缺什么、补齐后改变什么。不得只写一段“待确认”摘要。

## S6 定稿：强制章节

每篇 `*-领域知识定稿.md` 必须包含：

1. `## 概述与范围`
2. `## 不在本文展开`
3. `## 领域模型摘要`
4. `## 核心业务规则`
5. `## 关键明细`（当 S5 有 `## 结构化明细转交` 时）
6. `## 术语说明`
7. `## 待确认事项`
8. `## 溯源`

S6 只基于 S5。它可以提升可读性和中文表达，但不得从 S1/S3 新增 S5 没有承载的业务语义。

## S6 领域模型摘要

使用分层标签：

```markdown
## 领域模型摘要

- **一等业务对象**
  - <对象>：<读者向含义>
- **对象关系**
  - <关系>
- **状态机/状态转换**
  - <状态转换；没有稳定状态机时写“当前来源未形成稳定状态机”>
```

## S6 决策卡片

`## 核心业务规则` 下每个 `###` 小节必须使用以下卡片形态：

```markdown
### 规则 N：<读者向标题>

- **已确认规则**
  - <只承载一个判断点>
  - <另一个判断点>
- **待确认边界**
  - <边界；没有时写“当前来源未暴露额外边界”>
- **用户可见影响**
  - <一个可见影响>
- **关联待确认事项**
  - <无，或指向 ## 待确认事项 中的行动项>
```

规则：

- `已确认规则` 下的一级条目只承载一个判断点。不要用分号把多个分支、阶段、目标、奖励或展示后果压缩在同一条里。
- 规则依赖阈值、状态、字段、等级、文案等枚举明细时，卡片只写决策含义；可查询明细放入 `## 关键明细`。
- 交付/协作材料不得进入 `## 核心业务规则`。

## S6 关键明细

出现 `## 关键明细` 时，每个 `###` 小节都必须可查询：

- 有限行、读者需要逐项比较时，用 Markdown 表格。
- 规则化范围、模式或分组明细，用分层列表。
- 不得只写段落摘要。

## S6 待确认行动索引

`## 待确认事项` 必须始终包含这些分类标题，即使某分类暂时没有事项：

- `### 领域边界`
- `### 规则冲突`
- `### 数据与接口`
- `### 政策与展示`
- `### 待补充材料`

每条事项必须使用以下分层行动项形态：

```markdown
- **影响规则**：<影响的规则或主题>
  - **待确认/待补充**：<缺什么>
  - **建议确认人**：<确认角色>
  - **确认后影响**：<确认后会改变什么>
```

## S6 语言

- 正文以中文为主。
- 必要英文只允许作为：括号中的来源原名、反引号中的字段/API/系统锚点、已在 `## 术语说明` 解释的缩写，或 `## 溯源` 中的来源路径/链接。
- `## 术语说明` 使用 `中文术语（English Term）：解释`。
- 双语术语若出现在正文，正文第一次出现必须使用同样的双语锚点。
- 避免 `domain-knowledge/language/s6-reader-language-policy.json` 中定义的内部流程词和旧黑话。
