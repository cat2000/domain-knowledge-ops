# ticket_splitter_principles

## 0. 本文件怎么用（先读）

- **`ticket_system.md` 是运行时引擎的唯一输入**（拆单时模型不读本文件）。
- **本文件给人用**：拆单产出后，用它 **评判质量**，以及 **评审** 修改 `ticket_system.md` 的 PR。
- §3 的 **R1, R2, R2.5, R3, R4, R5, R6, R7, R8, R9, R10** 与 §5 **评审问题** 是与 `ticket_system.md` 共享的「好分解」词汇，不是第二层 prompt。
- **对话中的可选用法**：在后续轮次 @ 本文件，粘贴分解结果（或指向上一条 `ticket_system` 输出），请求对照评估（例如：对照 **§3** 与 **§5**）。不会自动触发；用户需要时再 @。

## 1. 目的

本文档定义工单拆分的 **评估 / 评审** 层。

帮助人回答：
- **已完成** 的分解是否根本正确
- 哪些是「核心质量」、哪些是 `ticket_system.md` 中的可选运行时细节
- 如何评审对 `ticket_system.md` 的 **拟议修改** 而不意外降低分解质量

**`ticket_system.md` 编码引擎的操作「怎么做」。本文件编码人检查输出时「好是什么样」。**

---

## 2. 范围

这些原则描述交付规划中 **分解项级** 输出的良好形态，供 **人** 评审。

用于评估（分解已存在后）：
- 分解是否根本正确
- 修改 `ticket_system.md` 是否会保持或削弱这些结果
- 分解是否偏向真实交付真相而非内部整齐

这些原则 **不** 定义：
- 输出格式细节
- 标题或字段的精确措辞
- 特定 prompt 的程序性指令
- 低层运行时启发式（除非表达核心原则）

---

## 2.5 评审优先级（人评估输出或编辑时）

评估 **已完成** 的拆分，或当临时用户文本、长模板、或「多拆几项 / 多铺面」压力与合理分解冲突时：

1. **INVEST / `Testable`（及 `source_requirement_note` 纠偏）**— 存在时 **优先于**「照搬 Jira 措辞」或「项数多看起来更好」。好的 `done_when` 与 **如何知道成功** 优先于填表。
2. **验证真相（R1）** 与 **扩项前证据充分（R2.5）** 比项数、模板饱满度、叙事 polish 更重要。
3. 分解不应仅因匹配外部模板、粒度 polish 或填格式而通过 — 先查 R1 与 R2.5。

---

## 3. 核心原则

### R1. 验证真相优先

分解必须首先服务工单的主验证面。

主验证面是建立真相的主要位置：
- 用户面
- 系统面
- 契约面

评审时确认分解 **在** 内部整齐、代码结构、实现便利 **之前** 优化主验证面。

### R2. 最小完整结果优先

默认保留最小完整结果。

可能是：
- 最小完整用户可验证体验
- 最小完整业务动作

若用户或下游系统只认组合行为为完整，不要机械拆成更小的技术或规则碎片。

### R2.5. 扩项前证据充分

输入证据不足以支持时，不要将分解扩成多个具体项。

额外项须至少有一条独立依据：
- 显式文本依据
- 独立验收依据
- 独立风险依据
- 独立状态推进依据

证据缺失时，合理拆分 **收敛** 为：
- 一个最小完整项
- 或一个 Spike + 一个最小 MVP 项

拒绝将常见工程考虑 **仅因 plausible** 就升格为独立项、却无独立依据的扩项。

### R3. 规模 override

完整性是默认，但不是绝对。

最小完整结果超过合理项大小时必须拆分。

此类拆分须保留有意义的验证，且不得退回任意技术切片。

优先 forced split 形态：
- 独立有意义的子体验
- 主路径先于边缘路径
- 有独立价值的 rollout / enablement 阶段
- 可独立验证的子动作

### R4. 真阶段链，非假阶段链

只有真实状态推进才 justify 串行阶段分解。

共享框架、模块、调度器或执行点 **本身** 不构成真阶段链。

串行依赖仅当后一片在前一片存在之前无效、不安全或不可测时成立。

### R5. 正交维度勿假串行

同一稳定基座上的独立规则维度不应被强塞进阶段链。

例如：
- 触发 cadence
- 数据窗口
- 停止条件
- 重试策略
- 展示分支
- 返回字段映射

共享基座基础设施不足以 justify 串行化。

### R6. 内部一致为次

内部归一化、状态统一、模型清理等 **不得** 压过外部可验证结果。

仅当：
- 它们 **本身** 是主验证面
- 或它们 **确实** 阻塞多个外部可验证切片且无法安全吸收

才可独立成项。

### R7. 不确定性须精确隔离

Spike 用于 ** materially** 改变分解、顺序、正确性或方案形态的不确定性。

Spike **不是**：
- 泛化 discovery
- 简单盘点
- 显然的实现任务
- 可安全吸收进非 Spike 项的工作

### R8. 可观察性约束精度

可观察性限制分解能有多精确且仍诚实。

高可观察性 → 可精确结构分解。
低可观察性 → 保守切片，禁止假精度。

评审时标记 **假装** 知道证据不支持的内部结构的拆分。

### R9. 依赖最小化

默认最小化依赖。

优先：
- 无依赖
- validation 依赖
- soft 依赖

仅当下游在上游完成前无效或不安全时用 blocking。

### R10. 用规则，不被规则用

好分解 **主动使用** 可用控制面以减少 blocking、缩小 blast radius、保持可合并性。

例如：
- feature flag
- adapter
- 兼容层
- fallback 路径
- 版本共存
- 可 mock 边界
- 双读/双写
- cohort rollout

**模式级示例（非穷尽）：** Strangler Fig、Branch by Abstraction、Anti-Corruption Layer (ACL)、Published Language、Open Host Service。这些 **映射到** 上述控制手段 — **不是** 单独的 **分解轴**；**不** 替代主验证面与最小完整结果逻辑。

这不是可选优化，是好分解判断的一部分。

---

## 4. 原则优先级

原则冲突时优先：

1. 验证真相优先
2. 最小完整结果优先
3. 扩项前证据充分
4. 规模 override
5. 真阶段链，非假阶段链
6. 正交维度勿假串行
7. 可观察性约束精度
8. 依赖最小化
9. 不确定性须精确隔离
10. 用规则，不被规则用
11. 内部一致为次

此顺序防止局部优化压过分解核心目的。

`ticket_system.md` 不重复此顺序；**两条原则看似冲突** 时，评审用此表决定哪条应赢，再判断拆分是否匹配（或记录为何不匹配）。

## 5. 评审问题

用以下问题评审分解或对 `ticket_system.md` 的拟议修改：

1. 分解是否识别正确的主验证面？
2. 是否保留最小完整结果？
3. 若扩成多项，每项是否有独立文本、验收、风险或状态依据？
4. 若拆最小完整结果，是否因规模 forced 而非便利？
5. 是否混淆共享基础设施与真阶段推进？
6. 是否不必要地串行化正交规则维度？
7. 是否让内部一致压过外部可验证价值？
8. Spike 是否用于真实不确定性，而非泛化调查？
9. 是否声称超过可观察性支持的结构精度？
10. 是否创建了本可为 validation 或 soft 的依赖？
11. 是否主动使用可用控制面？
12. 是否避免 **孤儿** prep（ payoff 仅是内部整齐或泛化「先领域模型/先重构」且 **未** 绑定 **本** 票可验证结果）— **同时仍允许** 有界 Tech Task (Enabler) 式 refactor/shaping，当 **当前复杂度确实阻塞** 直接加功能且项有 **清晰 scope 与可观察 `done_when`** 作为 unblocker？是否也避免 **机械** 一实体一项、碎片化 **最小完整可验证** 结果？（R2, R2.5, R6）
13. **`source_requirement_note`**（若使用）是否 **INVEST 主导**：是否 **先** 向 **Testable** **纠偏**（「完成」是什么、如何检查），再对齐条目，**差距与源文 cap**（如 dev-only、别处验证）**仅在其后**？若源文 prescribe 坏形态（如 dev vs testing 作为同一结果的主轴），输出是否 **避免** 镜像，且条目是否在证据限制内 **体现** INVEST 一致拆分？（R1, R2, R2.5）
14. 每项 **title** 是否读起来像 **单一主结果承诺标题**（一口气 refinement 扫读），而非顿号堆叠的 `scope` dump？（presentation P11；title 质量 ≠ 自动拆项）
15. 每项 **`scope` bullet** 是否陈述 **一条可口头扫读的 concern**，而非用 `：`/`；`/`/` 焊 tab/filter/card 成一条 inventory？（presentation P12；改 bullets，非自动拆 Story）

## 6. 与 `ticket_system.md` 的关系

| 产物 | 角色 |
|------|------|
| `ticket_system.md` | 引擎产 split 的 **唯一** prompt。 |
| `ticket_splitter_principles.md`（本文件） | split 产出 **之后**，人用来 **评估质量** 与 **评审** `ticket_system.md` 变更。 |

- **评审 PR** 改 `ticket_system.md` 时，用本文件与 §5 确保编辑不削弱你在意的结果（§3 的 R 标签，含 **R2.5**）。
- 引擎 **不读** 本文件；生成时没有「原则 override 运行时」。**评审** 时仍可说：「此输出差因违反 R2.5 / R4」，必要时再改 `ticket_system.md`。
- **命名对齐：** `ticket_system.md` 内 `(R#)` 标签是本地助记，与 §3 **R1, R2, R2.5, R3, …, R10** **概念对齐**，便于 reviewer 与作者共用语言。
- **联合用户行为（最小用户可验证体验）：** `ticket_system.md` **DEFINITIONS** 中，若干条件/控制/动作可构成 **一个** **最小用户可验证** 体验，即 **R2**（最小完整结果）、**R4**（仅真推进）、**R5**（正交维度不假串行）在运行时的体现。对照该文本评审拆分时，也查 **`ticket_system.md` Step 4 — thin-slice `Avoid` 列表**（如 groundwork-only first、standalone normalization、FE/BE-only splits）与 **Step 7**（规模与 anti–over-split，含 condition/frequency/action **共同** 定义「完成」时勿碎片化）。
- **R10 与 DDD / 集成模式名：** **R10** 下列出的模式名（Strangler Fig、Branch by Abstraction、ACL、Published Language、Open Host Service）**说明** 与 `ticket_system.md` **Step 5** 中具体手段相同的 **控制** 思想。它们是 **何时** 应用那些手段的 **触发与词汇** — **不是** 如何 *计数* 或 *排序* 项的主轴。措辞与 Step 5 **对齐**：用模式 **回忆** 控制面，**不要** 用模式驱动拆分。
- **INVEST 纠偏 vs 源文：** 运行时 **`Primary intent`** 把 **INVEST**（尤其 **Testable**）放第一；**`source_requirement_note`** 是主 **纠偏** 通道。对 dev-only/别处验证 cap 的保真性是 **约束**，不能替代源文允许范围内每项 **可观察** 的 `done_when`。**代码**（用户/会话提供时）可供给 route/package/service **锚点** — **不要** 发明模块名。评审见 **§2.5** 第 1 条、**§5** Q13。
- **Backlog 类型映射（人的流程）：** 运行时输出应已用三类：User Story、Tech Task (Enabler)、Spike。分类：用户可见价值切片 → User Story；纯迁移/重构/兼容 → Tech Task (Enabler)；不确定性隔离 → Spike。
