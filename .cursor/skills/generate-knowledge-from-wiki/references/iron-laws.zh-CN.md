# generate-knowledge-from-wiki · 铁律（按需加载）

> English SSOT: [`iron-laws.md`](./iron-laws.md).
>
> Agent：仅在执行 Compose / 争议裁决时阅读本文件；日常备料以 `SKILL.md` + `RUNBOOK.md` 为准。

## 铁律

0. **Strategy-first 硬停**：若 `domain-knowledge/s2-domain-profiles.json` 的 `checklist_themes` 为空，或 `strategy.md` §2 仍大量「待填」——**立即停止**，引导 `@setup-domain-ops`（填实 strategy → 派生 profiles → 人确认）。**禁止**静默套用任何仓库预置业态模块。
1. 默认 **备料**：**Ingest + Recognize**（**S1 → S2**）→ **暂停**（`DOMAIN_MODULE_CHECKLIST.md`）；**S1–S6 禁止翻译**
2. **成稿 Compose**（**`确认`** 后 **`继续`**）：**S3 → S4 → S5 → S6 → S7**（仅 `确认` 行）
3. **S3 只做保真分轨、索引与可验证传递，S4 建领域模型，S5 基于模型生成工作稿且语言与源一致，S6 产出原语言裁决定稿（不翻译），S7 仅做目标语言（`deliverable_locale`）表达转换**（不得引入新语义）
4. **`确认`** = 认可领域块划分，授权 Compose；**≠** 已完成定稿（需 S6 + S7）
4a. **零 tagged 来源不得确认**：closure 中该 slug 无 Confluence/Jira 页（S2 备注 / `tagging_acceptance.py` / `pages_with_props=0`）时保持 **待确认**。禁止为「勾完清单」而确认。
4b. **确认前打标验收**：S2 后运行 `tagging_acceptance.py --root-id <R>` 并展示报告；禁止引导「全部确认」。行业切块只是题面；完整性靠双向打标 + 报告。已配置 board 但 Jira attribution=0 = 打标不完整，应先补 Jira。
4c. **成稿穷尽度**：S3 后跑 `tagging_acceptance.py --after-s3`。规则数远小于 `pages_with_props` → 写入待确认或不得声称「已齐」。低证据 S7 必须带 **证据不足** 横幅（对 risk 非 SSOT）。
5. S1 `facet-*` = 机器粗分，**≠** checklist 确认 slug（命题级认域在 **S2**）
6. 正本在 Confluence；勿长期手改 `materialized/` 当权威
7. **S1 完整性铁律**：抽取报告有 page error 时默认不得进入 S2；只有显式 `--allow-partial` 才可写 partial handoff，且必须把缺页风险带入后续判断
8. **S1 物化清单铁律**：`materialized/` 是当前 source set 的生成物；每次物化必须写 manifest 并清理不再属于当前源集合的旧 `.md`
9. 成稿产物在 `curated/by-root/<R>/`（`_deliver/`、`_aggregate/`）
10. 禁止仓库 HTTP LLM API
11. **S5 规则链必须可决策，S6 核心规则必须可读可裁决**：S5 写清“对象 + 条件 + 用户可见影响 + 具体锚点”；S6 以读者向语言承接这些语义，不使用字段模板兜底。
12. **禁止模板句过门禁**：如“进入业务场景即生效”“以后续实现为准”等空泛条文，视为不合格定稿
13. **S6 产品化铁律**：S6 是读者向知识产品，不是规则字段模板；核心业务规则必须写成读者决策卡片，稳定呈现 `已确认规则 / 待确认边界 / 用户可见影响 / 关联待确认事项`。每个标签必须作为独立粗体列表项，具体条件、分支、时间窗、例外、可见后果和来源语义放入缩进子项；同一条一级 bullet 不得用分号压缩多个分支、阶段、目标、奖励或展示后果，必须拆成分层子项。
14. **S6 双语术语锚点铁律**：S6 主文必须中文为主；凡在 `## 术语说明` 声明为 `中文术语（English Term）` 的关键业务术语，如果正文使用该中文术语，正文首次出现必须采用同样的 `中文术语（English Term）` 锚点，后续可只用中文。必要英文只允许作为括号内来源原名、字段/API/系统名、业务专有缩写或来源链接/路径出现；字段/API/系统名必须用反引号作为锚点；业务专有缩写必须在 `## 术语说明` 解释，且不得用英文替代中文叙述。
15. **S6 待定问题索引铁律**：`## 待确认事项` 不是备注列表，必须写成可行动索引；每条待确认事项必须以 `影响规则` 作为主列表项，并在缩进子项中包含 `待确认/待补充`、`建议确认人`、`确认后影响`。
16. **S6 分层阅读铁律**：`## 领域模型摘要` 的一等业务对象、对象关系、状态机/状态转换必须用粗体标签加缩进子项表达；多对象、多关系、多状态不得压缩在同一长句或同一列表项中。
17. **S6 结构化明细保真铁律**：当 S1/S5 存在映射表、状态枚举、字段清单、公式/阈值表、时间窗表或可见文案清单时，S6 不得只摘要；必须在 `## 关键明细` 或对应规则卡片的 `明细` 子块中保留可查明细。若明细因来源不完整不能承诺，必须进入 `## 待确认事项`，说明缺什么和补齐后影响。
18. **S6 读者语言精修铁律**：S6 禁止残留流程阶段语言、内部验收语境和旧黑话；具体表达形态由 `domain-knowledge/language/s6-reader-language-policy.json` 配置，脚本只做通用检查，不写业务域规则。
19. **S6 强制结构**：必须包含 `## 概述与范围`、`## 不在本文展开`、`## 领域模型摘要`、`## 核心业务规则`、`## 术语说明`、`## 待确认事项`、`## 溯源`；若存在 `no-text/待补充`，必须进入 `## 待确认事项` 的分类小节。
20. **跨领域通用层**：S3 必产出 `propositions` / `命题清单`；`decision-atoms` 与 `conflict-ledger` 仅作为 `propositions` 的最小派生审计视图，不生成 `policy-primitives` / `policy-graph`
21. **S2 primary facet 复核铁律**：S1 `facet-*` 只是粗分默认值；即使命中 `primary_facet_to_slug`，S2 仍必须用页面标题/路径级 `route_overrides` 复核跨 slug 归属。只有标题/路径明确命中其它 confirmed slug 时才转交，正文偶发词不得触发跨 slug 改派。
22. **S3 cross-slug handoff 铁律**：S2 发生 primary facet 跨 slug 转交时，S3 必须生成 `_aggregate/CROSS_SLUG_HANDOFF.json|md`，作为 S4/S5 的审计输入；Agent 必须检查该文件，确认转交页是否已被目标 slug 吸收、降层或排除。
23. **S3 source set 铁律**：S2 `_materialization_closure.json` 是 Compose source set 的唯一真相；S3 只解析 closure 中指向已确认 slug 的 source，不重新按 `facet-*`、Jira attribution 或目录扫描决定准入。
24. **S3.5 结构化要求**：`proposition_items` 优先输出 `decision_block + decision_confidence`；S4 应以高置信度决策块重挂载，而非句子拼接。
25. **S3 表格型来源标记铁律**：当 S1 出现长映射表、字段表、状态枚举表、资格/等级/职称/奖励档位表、时间窗表或可见文案表时，即使单行不构成完整命题，S3 也必须在页面级标记结构化来源信号，并保留表头/字段、密集行数量和样例行，供 S5 转交；禁止因“未抽取到明显命题”让长表静默消失。
26. **S4 主输入铁律**：Agent 必须先读 `_aggregate/<slug>/<slug>-propositions.json`、`*-命题清单.md` 与 `_aggregate/CROSS_SLUG_HANDOFF.md`；`decision-atoms` 只能辅助核对，禁止作为 S4 生成主源或投影模板。
27. **S4 领域模型铁律**：S4 必须先产出 `## 领域模型`，并按层分开识别 `一等业务对象`、`指标/字段`、`展示容器`、`对象关系`、`状态机/状态转换`、`边界候选`；UI/API/字段/指标/边界材料不得与一等业务对象平级。
28. **S4 模型门禁铁律**：`domain_model_quality.py` 必须验证 S4 分层模型、S5 链对象继承 S4 `一等业务对象`、字段/API/页面降层；脚本只验证显式表达，不生成 S4/S5 正文。
29. **S5 规则链挂载铁律**：S5 工作稿中的每条闭环链必须挂到 S4 分层领域模型；链内 `领域对象` 必须来自 `一等业务对象`，指标/字段/公式/API token/展示容器只能写入 `展示容器/字段锚点`，不得冒充领域对象。
30. **S5 处置铁律**：Agent 必须逐类处置 `contract_candidate` / `evidence_note` / `noise_context`：进入闭环链、进入待裁决、作为支撑证据、或排除为噪声；不得无解释丢弃高价值 evidence。
31. **S5 evidence 提升铁律**：`evidence_note` 不是低优先级材料；凡含公式、阈值、状态枚举、展示映射、卡片可见性、时间窗、资格/奖励/金额后果、用户可见字段的 evidence，Agent 必须判断为闭环链、半闭环链或待裁决问题，不得仅因 `candidate_type != contract_candidate` 降级为支撑材料。
32. **S5 语义重挂载主责**：Agent 必须基于 `strategy.md`、`distill-quality-bar.md` 与 S4 领域模型判断哪些源语义是一等业务对象；凡影响对象、条件、业务分支、状态/阶段/等级、资格、奖励、展示规则、阈值、时间窗或用户可见影响的内容，必须进入闭环链或待裁决区。
33. **S5 半闭环拆分铁律**：当 S1/S3 已明确业务规则但实现/API/字段/检测方式未明确时，禁止整体升格或整体降级；明确业务规则进入闭环链，未明确实现/字段/检测进入待裁决区，闭环链必须标注 `关联待裁决`。
34. **S5 命名业务结构保护**：源材料中的命名分支、阶段、路径、档位、等级、状态、周期、市场、卡片等，只要承担业务判定或可见后果含义，就不得被抽象成无名“条件 A/B/C”后消失；可归一命名，但必须保留业务含义。
35. **S5 语义归一铁律**：一条 S5 链只对应一个业务判定对象；页面、API、字段、UI 卡片只是证据/展示位置/字段锚点。相同业务动作出现在多个页面或 UI 容器时，必须合并为主规则，并列出展示位置或证据来源，禁止重复成链。
36. **S5 顺序归一铁律**：S1/S3 保留来源顺序；S5 必须判断同页顺序是否代表业务执行顺序，并按 `准入/资格 -> 前置校验 -> 核心判定/计算 -> 结算/状态落账 -> 展示/运营 -> 占位补录` 重组。若保留来源顺序，必须说明其业务含义；S6 禁止再裁决顺序。
37. **S5 组织顺序计划铁律**：S5 必须在正文链之前写 `## 组织顺序说明`，按最终落稿顺序逐条列出 `链 N：标题 — 顺序理由`；`## 已闭环决策链` 的链号和标题必须与该计划一致，禁止只声明顺序归一但正文不执行。
38. **S5 全稿顺序一致铁律**：`领域概述`、`输入处置摘要`、`领域模型`、`组织顺序说明`、`已闭环决策链`、`待裁决关键问题` 必须共享同一条业务主线；`## 组织顺序说明` 是链序唯一真相，输入处置摘要只说明“已按组织顺序重排”，禁止另写一套可能不一致的详细链序。
39. **S5 待裁决互链铁律**：闭环链写 `关联待裁决：问题 N` 时，`问题 N` 必须存在，且问题块必须反向写 `关联链：链 X` 包含该链；待裁决问题必须写 `关联链：链 N` 或 `关联前置域：...`，并说明 `决策影响`。
40. **S5 领域边界降层铁律**：排除材料、噪声、工程协作、发布/合规支撑、待迁移主题、待归属材料不得出现在 `## 领域模型` 的 `领域对象` 中；只能写入输入处置摘要、待裁决前置域或排除说明。
41. **S5 模型分层铁律**：指标、进度、金额、公式、API 字段、页面卡片只能放在 `指标/字段` 或 `展示容器`；`对象关系` 必须说明对象之间的依赖/承载/计算/展示关系；`状态机/状态转换` 必须说明核心状态如何变化。
42. **S5 模型-链一致铁律**：链的 `领域对象` 不得是指标、字段、公式、金额、进度、目标、页面、卡片、列表或 API；待裁决若阻塞指标口径，必须说明所属一等业务对象。
43. **S5 工作稿门禁铁律**：`s5_work_draft_quality.py` 必须验证闭环链必填字段、组织顺序一致、待裁决互链、未决词位置和结构化明细形态；脚本只验证显式表达，不生成 S5 正文。
44. **S5 强制结构**：必须分为 `## 输入处置摘要`、`## 领域模型`、`## 组织顺序说明`、`## 已闭环决策链` 与 `## 待裁决关键问题`；输入处置摘要必须说明 high-value evidence 的提升/待裁决/排除理由，以及重复/跨容器语义如何归一；`未确定/待确认/TBD` 必须进入待裁决区，不得留在闭环区。
45. **S5 结构化明细转交铁律**：当 S3 proposition、S3 页面级结构化来源信号或 S5 语义重挂载发现映射表、状态枚举、字段清单、公式/阈值表、时间窗表、可见文案清单或编号/档位/等级映射时，S5 必须新增 `## 结构化明细转交`；先按“完整展开 / 规则化压缩 / 待确认”三分法裁定粒度，再按业务规则分 `###` 小节输出可查询明细。该区是 S6 `## 关键明细` 的唯一来源，S6 不得回看 S1 补明细。
46. **S5 明细粒度三分法铁律**：读者需要按编号、状态、字段、时间窗、职级、奖励档位或可见文案逐项查询/判断时必须完整展开；来源行可由稳定公式、连续编号、等差阈值或同构命名无损生成，且单行没有例外含义时才可规则化压缩，并必须保留范围、公式、端点与例外；来源冲突、不完整、只给设计稿占位、字段含义未闭环或是否用户可见未确认时，不得伪完整展开，必须进入待确认事项并说明补齐后影响。
47. **S5 禁止伪闭环**：闭环链禁止占位词（如“待补充/按来源条件触发/按来源动作/待确认对象”）；同一规则簇只承载一个业务判定问题，禁止把不同判定问题混并。
48. **S5 来源覆盖**：S3 `propositions` 中高贡献 contract/evidence 来源页必须在 S5 工作稿引用；缺失视为不合格。
49. **S5/S6 作者契约铁律**：S5/S6 落稿前必须先读 `domain-knowledge/distill-authoring-contract.md`；该文件是可通过门禁的最小 Markdown 作者契约。Agent 仍负责语义裁决，脚本只验证契约是否被显式表达。
50. **约束来源原则**：上述规则以 Skill + `strategy.md` + `distill-quality-bar.md` + `domain-knowledge/distill-authoring-contract.md` 为主定义，脚本门禁仅做验证实现，不得反向主导语义标准。
51. **S2 先停闸再成稿（硬门禁）**：`DOMAIN_MODULE_CHECKLIST.md` 仍含 `待确认` 时，禁止进入 S3；默认由脚本阻断（除显式 override）
